================================================================================
                     BEAVER'S CHOICE PAPER COMPANY
                      MULTI-AGENT SYSTEM DESIGN NOTES
================================================================================

OVERVIEW
--------
The system uses smolagents to create a multi-agent workflow for a paper supply
company. Customers send text requests for paper supplies, and the system
processes them through a coordinated 5-agent architecture with deterministic
business logic for quoting, stock validation, and transaction recording.


AGENT ARCHITECTURE
------------------

The system defines 5 agents using the ToolCallingAgent class:

1. ORCHESTRATOR AGENT (Main)
   - Purpose: Central coordinator — drives the full pipeline from raw customer
              text to final formatted response. Business rules are enforced in
              code, not in LLM prompt instructions.
   - Tools: extract_request_items, get_catalog_item_details, check_inventory,
            check_item_stock, get_delivery_date, get_cash, get_financial_report,
            create_sale_transaction, create_stock_order, search_historical_quotes

2. INVENTORY AGENT
   - Purpose: Manages all stock-level queries and supplier replenishment orders.
              Enforces cash sufficiency before any stock purchase.
   - Tools: check_inventory, check_item_stock, get_delivery_date, create_stock_order

3. QUOTING AGENT
   - Purpose: Handles catalog lookups, unit pricing, and historical quote search
              to support accurate, context-aware quote generation.
   - Tools: get_unit_price, get_delivery_date, search_historical_quotes,
            get_catalog_item_details

4. FULFILLMENT AGENT
   - Purpose: Processes and records validated sale transactions. Enforces
              catalog existence, stock availability, and quantity checks
              before any database write.
   - Tools: create_sale_transaction, check_item_stock

5. REQUEST EXTRACTION WORKER  (Dynamic 5th Agent)
   - Purpose: Lightweight LLM agent with NO tools. Instantiated fresh per
              request inside extract_request_items(). Parses free-text
              customer requests into structured JSON order lines.
   - Tools: None
   - Note: Shown with dashed border in architecture diagrams to indicate it
           is dynamically instantiated, not a static module-level instance.


MODEL CONFIGURATION
-------------------

The system uses OpenAIServerModel with the Vocareum proxy:

    model = OpenAIServerModel(
        model_id="gpt-4o-mini",
        api_base="https://openai.vocareum.com/v1",
        api_key=openai_api_key,
    )

Note: Use "api_base" not "base_url" — the latter causes an error with smolagents.
All 5 agents share the same model instance.


TOOL DEFINITION
---------------

All tools must be decorated with @tool and include proper docstrings.
The docstring MUST include an Args: section for each parameter, otherwise
smolagents will raise a DocstringParsingException.

Key tools added in the updated design:
  - extract_request_items()    : Invokes RequestExtractionWorker to parse text
  - get_catalog_item_details() : Returns normalized name, in_catalog, stock, price
  - item_exists_in_catalog()   : Internal guard used by sale and stock order tools
  - get_cash()                 : Returns current cash balance for a given date


PROCESSING FLOW
---------------

When a customer request comes in:

1. process_request(request_text, agent_date) is called
2. extract_request_items() triggers the RequestExtractionWorker (5th agent)
   to parse the raw text into: {items: [{requested_name, quantity}]}
3. _coerce_request_items() converts JSON into RequestItem dataclasses
4. build_quote() is called deterministically:
   a. normalize_item_name() maps customer wording to catalog names
   b. get_catalog_item_details() checks availability and price
   c. Fulfillable qty = min(requested_qty, available_stock)
   d. Line total = fulfillable_qty x unit_price
   e. 10% discount applied if fulfillable subtotal > $500
   f. search_historical_quotes() fetches similar past orders for context
   g. get_delivery_date() estimates delivery based on max requested qty
5. fulfill_quote() calls create_sale_transaction() for each fulfillable line
   (catalog, stock, and quantity validations enforced inside the tool)
6. render_customer_response() formats the final markdown table + summary
7. The formatted string is returned to the caller

Key design principle: Business rules are enforced deterministically in Python
code — NOT left to prompt instructions or LLM judgment.


DATACLASSES
-----------

Three dataclasses structure the quoting pipeline:

  RequestItem(requested_name, quantity)
    Represents a single parsed line from the customer request.

  QuoteLine(requested_name, normalized_name, quantity, unit_price, line_total,
            in_catalog, available_stock, fulfillable_qty, status)
    Represents one item line in the quote with full pricing and availability.

  QuoteResult(lines, subtotal, discount, total, estimated_delivery_date,
              historical_context)
    Aggregates all quote lines with totals, discount, and supporting context.


DATABASE LAYER
--------------

The system uses SQLite (paper_company_db.db) with these tables:

  - inventory      : catalog items with unit prices and min stock levels
  - transactions   : logs all stock_orders and sales with dates and amounts
  - quotes         : historical quote data with explanations and metadata
  - quote_requests : incoming customer inquiries

Utility functions:
  - get_all_inventory(date)            : Returns dict of {item_name: stock}
  - get_stock_level(item, date)        : Returns DataFrame with current_stock
  - get_cash_balance(date)             : Returns float cash position
  - get_supplier_delivery_date(d, qty) : Estimates delivery; raises ValueError
                                         on invalid date (no silent fallback)
  - search_quote_history(terms, limit) : Searches historical quotes; LIMIT
                                         is clamped to 1-20 (no SQL injection)
  - create_transaction(...)            : Uses db_engine.begin() context so
                                         last_insert_rowid() is reliable
  - generate_financial_report(date)    : Returns cash, inventory_value,
                                         total_assets, top_selling_products


ITEM NORMALIZATION
------------------

normalize_item_name(requested_item) maps free-text names to canonical catalog
names using a two-stage lookup:
  1. Exact / substring match against ITEM_MAPPING dictionary
     e.g. 'washi tape' -> 'Decorative adhesive tape (washi tape)'
  2. Substring match against paper_supplies catalog as fallback

item_exists_in_catalog(item_name) verifies the resolved name is a real catalog
entry. Unmatched items are flagged as unavailable in the quote line.


DISCOUNT POLICY
---------------

  - 10% discount applied when the fulfillable subtotal exceeds $500.
  - Calculated on fulfillable qty totals only (not full requested qty).
  - No discount applied to out-of-stock or catalog-unavailable lines.


DELIVERY LEAD TIMES
-------------------

Supplier delivery is deterministic based on quantity:
  -    1-10  units : Same day (0 days)
  -   11-100 units : 1 day
  - 101-1000 units : 4 days
  -   1000+  units : 7 days

Invalid date input raises ValueError instead of silently falling back.


TESTING
-------

Run test scenarios with:
    python project_final.py

This processes quote_requests_sample.csv, sorts by request_date, and outputs
results to test_results.csv. The system should demonstrate:
  - At least 3 requests with cash balance changes
  - At least 3 successfully fulfilled quote requests
  - Partial fulfillment cases (stock covers only part of requested qty)
  - Some requests not fulfilled (out of stock or item not in catalog)
  - A final financial report printed to console
